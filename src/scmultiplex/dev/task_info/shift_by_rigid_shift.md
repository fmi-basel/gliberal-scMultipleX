### Purpose

* Copy one or more 2D or 3D label images from a reference OME-Zarr image to a moving-round OME-Zarr image.
* This allows segmentation generated only in the reference round to be propagated to subsequent moving rounds using the previously calculated rigid registration.
* Apply a previously calculated 2D rigid transformation consisting of XY rotation and translation.
* Load the rigid transform from a `sequence.json` file stored in the moving image’s `registration/{registration_name}` folder.
* Optionally load the transform from a related OME-Zarr container, such as a 2D MIP container, by modifying the input Zarr and image names with configurable suffixes.
* Apply the same XY transformation independently to every Z slice of a 3D label image.
* Process 3D labels in Z blocks so that only a limited number of full XY slices are loaded into memory at once.
* Use nearest-neighbor interpolation to preserve integer label IDs.
* Pad the reference label with background value `0` when necessary to accommodate differences in XY image extent.
* Support copying multiple labels in one task invocation.
* Optionally assign each shifted label a new output name.

### Inputs

* `label_name_to_shift`: List of label names to copy from the reference image and transform.
* `registration_name`: Name of the registration subfolder containing `sequence.json`. Defaults to `rigid_2D`.
* `load_registration_from_other_container`: If `True`, construct the registration-container path from the moving-image path using the configured suffix arguments. If `False`, load the registration directly from the current moving OME-Zarr container.
* `zarr_suffix_to_add`: Optional suffix added to the OME-Zarr container name when locating the registration container. Defaults to `_mip`.
* `image_suffix_to_remove`: Optional suffix removed from the image name when locating the registration container.
* `image_suffix_to_add`: Optional suffix appended to the image name when locating the registration container.
* `new_shifted_label_name`: Optional list of output label names. The names must follow the same order and have the same length as `label_name_to_shift`. When omitted, each output label keeps its original name.
* `overwrite_label`: If `True`, allow an existing output label in the moving OME-Zarr image to be overwritten.

### Outputs

* A transformed label image saved in the moving-round OME-Zarr container for every label listed in `label_name_to_shift`.
* The output label keeps its original name unless a corresponding name is supplied through `new_shifted_label_name`.
* A masking ROI table named `{output_label_name}_ROI_table` saved in the moving-round OME-Zarr container.
* When a new output label name is supplied, a corresponding masking ROI table with the same new name is also created in the reference-round OME-Zarr container.
* A multiscale label pyramid generated for each transformed output label.

The task does not calculate or save the rigid transformation. It loads a transform previously generated by the rigid-registration task from:

```text
{registration_zarr_url}/registration/{registration_name}/sequence.json
```

The stored transformation is converted from physical units to pixel coordinates before being applied to the labels.

### Transformation behavior

* The transformation is applied in the XY plane only.
* For 3D labels, the same 2D transformation is applied independently to every Z slice.
* `scipy.ndimage.affine_transform` is used with nearest-neighbor interpolation (`order=0`) to avoid introducing interpolated label values.
* Pixels outside the source-image extent are filled with label value `0`.
* The transformed label is adjusted to match the destination image’s XY dimensions before being written.
* The task verifies that the moving image and the image used to calculate the registration have matching X and Y pixel sizes.
* Label arrays are expected to have shape `(Z, Y, X)`. A 2D label is expected to be represented as `(1, Y, X)`.

### Limitations

* Supports rigid 2D Euclidean transformations only: rotation and XY translation.
* Does not support scaling, shear, nonlinear deformation, or local warping.
* Does not perform any Z translation, Z rotation, or other transformation along the Z axis.
* Assumes that applying one XY transform across all Z slices is appropriate for the dataset.
* The transformation must already exist and must be stored in the expected `sequence.json` format.
* The label image and the image used to calculate the registration must share the same XY pixel size.
* Because each transform may move pixels across chunk boundaries, XY chunks cannot be transformed independently. Full XY slices are therefore loaded and processed in Z blocks. This leads to slower processing times.
